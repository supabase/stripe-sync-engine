# Idea: CLI Specification Format

## Context

OpenAPI is the standard for describing HTTP API inputs and outputs deterministically.
There is no equivalent standard for CLIs — but the input side (commands, flags, args) is
well-solved by several frameworks. The output side (structured response shape) remains
largely unsolved unless the CLI commits to emitting JSON.

## Relevant Art

### Fig / Amazon Q autocomplete specs
TypeScript objects describing every subcommand, flag, arg type. Plain TS — fully extensible.
Originally built for terminal autocomplete (withfig/autocomplete), now maintained less
actively under Amazon Q. Stripe CLI has a spec at:
`github.com/withfig/autocomplete` → `src/stripe.ts`

Key features:
- `isVariadic`, `isDangerous`, `template: "filepaths"`, `suggestions` (inline enums)
- Reusable arrays spread across commands (`...globalOptions`)
- `generators` field for dynamic runtime suggestions (runs a subcommand to fetch completions)

### oclif manifest (`oclif.manifest.json`)
Generated from TypeScript command classes. Machine-readable JSON describing all commands,
flags (type: `"option"` | `"boolean"`), args, defaults, required/optional, aliases.
Can be extended via static properties on command classes.

### Others
- **docopt** — spec is the help string itself; simple but limited
- **Typer (Python)** — type hints are the spec; emits JSON schema via `typer utils docs`
- **clap (Rust)** — derive macros; richest type system; generates completions/man pages
- **Smithy (AWS)** — IDL targeting both HTTP and CLI from one model

## Relevance to Sync Engine

The sync-engine service CLI is already derived from the OpenAPI spec via `createCliFromSpec`.
This means the **input spec already exists** — it's the OpenAPI schema.

Gaps:
1. The derivation from OpenAPI → CLI is implicit; no standalone CLI spec is committed
2. Output is unstructured text for most commands
3. No way for external tools to introspect available commands without parsing OpenAPI

## Potential Directions

- **Emit a Fig-style spec** from the OpenAPI schema at build time — gives completions + a
  machine-readable input spec independent of HTTP concepts
- **Commit to JSON output** for all commands (`--output json`) and JSON Schema the shapes
- **oclif migration** if the CLI grows complex enough to warrant first-class plugin support
  and a generated manifest

## Status

Idea only. No implementation planned.
